## 1. プロトコル全体構造 (確定)

本家 `esphome/components/daikin_arc` と同じタイミング・ヘッダバイトで、そのままデコードできる。

- キャリア周波数: 38kHz
- 1回の送信は **2つのフレーム**から成る(間に `mark(BIT_MARK) + space(MESSAGE_SPACE≈35ms)` が入る):
  1. **header frame (20バイト)** — 先頭に `pre_mark(≈10000us) + pre_space(≈-25100us)` の
     長いリーダーが付く
  2. **state frame (19バイト)** — 現在の運転状態(モード・温度・風量など)を表す本体。
     こちらにはリーダーは付かず、いきなり `header_mark(≈3450us)+header_space(≈-1760us)` から始まる。
- 各バイトは LSBファースト、`bit_mark(≈400us)` + `one_space(≈1300us)` / `zero_space(≈480us)`。
- チェックサムは両フレームとも「最終バイト以外の総和を256で割った余り」= 単純8bit加算。
  実測分は全てchecksum一致を確認済み。
- 両フレームとも先頭4バイトは固定ヘッダ `11 DA 27 00`。

タイミング定数は `daikin_arc472a66.h` の `DAIKIN_*` 定数のままで実測と一致している
(`decode_daikin_arc.py`の`TIMING`と同じ値)。ここは変更不要。

### 2-1. モードビット (byte[5]) — **全モード確定**

5モードすべてを個別ボタン(冷房・除湿・暖房・送風・快適エコ自動)で直接実測し、
既存コンポーネントの定数と**完全一致**した。

| モード | byte[5] 実測値 | 内訳 | 定数 |
|---|---|---|---|
| 冷房 | `0x39` | `0x01`電源 \| `0x30`冷房 \| `0x08`固定 | `DAIKIN_MODE_COOL=0x30` ✓確定 |
| 除湿 | `0x29` | `0x01`電源 \| `0x20`除湿 \| `0x08`固定 | `DAIKIN_MODE_DRY=0x20` ✓確定 |
| 暖房 | `0x49` | `0x01`電源 \| `0x40`暖房 \| `0x08`固定 | `DAIKIN_MODE_HEAT=0x40` ✓確定 |
| 送風 | `0x69` | `0x01`電源 \| `0x60`送風 \| `0x08`固定 | `DAIKIN_MODE_FAN=0x60` ✓確定 |
| 快適エコ自動 | `0x09` | `0x01`電源 \| モードビットなし \| `0x08`固定 | `DAIKIN_MODE_AUTO=0x00` ✓確定 |

固定ビット`0x08`は全モード共通で立っている(本家と同じ)。停止時は電源ビット(`0x01`)だけ
落として他は直前値を保持(4章参照)。**`daikin_arc472a66.h`の`DAIKIN_MODE_*`定数は修正不要、
コメントの「TODO: 実測して確認」は外してよい。**

なお「暖房」ボタンには二度押しで入る「高温風」モードもあるが未実測(優先度低)。
「快適エコ自動」ボタンは連続で押すと**快適エコ自動(byte14=0x0a) ⇔ ビズ自動(byte14=0x0e)の
2状態をトグルする**ことを確認した(2-4節参照)。3つ目の状態はなさそう。

### 2-2. 温度 (byte[6]) — **冷房・暖房・送風は確定、自動系は要検証**

式: `温度 = byte[6] >> 1`, 下位ビット1なら+0.5℃。

| モード/状態 | 表示温度 | byte[6] | 検算 |
|---|---|---|---|
| 冷房 | 28℃ | `0x38` | `56>>1=28` ✓ |
| 暖房 | 23℃ | `0x2e` | `46>>1=23` ✓ |
| 送風 | (温度概念なし) | `0x32` | 本家の固定値と一致(実測で確認) |
| 除湿 | (絶対温度なし) | `0xC0` | 固定値。下位は湿度用(2-6参照) |

→ **冷房・暖房で同じ一次式が成立することを確認した。式自体は確定としてよい**(離散点は
28℃/23℃の2点のみだが、他機種でも共通の実装なので信頼度は高い)。境界値(18℃/32℃付近)は
未実測だが優先度は低い。

**快適エコ自動モードだけ特殊**: 絶対温度ではなく「適温からの相対オフセット」表示。**式が確定した**:

| オフセット | byte[6] |
|---|---|
| -1.0℃ | `0xDE` |
| -0.5℃ | `0xDF` |
| 0(適温、未調整のデフォルト) | `0xC0` |
| +0.5℃ | `0xC1` |
| +1.0℃ | `0xC2` |
| +1.5℃ | `0xC3` |
| +2.0℃ | `0xC4` |

**式**: オフセット`n`(℃、0.5刻み)として、
- `n >= 0` のとき `byte6 = 0xC0 + n*2`
- `n < 0`  のとき `byte6 = 0xE0 + n*2` (= `0xE0 - |n|*2`)

0を境にプラス側/マイナス側で切片が`0xC0`/`0xE0`と異なる2分岐の式(単純な符号付き整数ではない)。
6点のデータで矛盾なく一致しているので確度は高い。可変範囲(何度まで±できるか)は未確認だが、
実装上の優先度は低い。

### 2-3. 風量 (byte[8]) — **確定**

風量ボタンを「しずか→1→2→3→4→5→自動→しずか」と1周させて全段階を実測、
既存コンポーネントの定数と完全一致した。

| 表示 | byte[8] |
|---|---|
| 1 | `0x30` |
| 2 | `0x40` |
| 3 | `0x50` |
| 4 | `0x60` |
| 5 | `0x70` |
| 自動 | `0xA0` |
| しずか | `0xB0` |

→ `DAIKIN_FAN_1`〜`DAIKIN_FAN_5` / `DAIKIN_FAN_AUTO` / `DAIKIN_FAN_SILENT` は**修正不要**。
なお「しずか」はスイング関連ボタンとは別の、**風量ボタンのサイクルの一部**であって
独立した「静音ボタン」ではない(このリモコンには節電/econoボタンも見当たらないとのこと)。

### 2-4. byte[13] は「特殊運転フラグビット」の集合 — **確定**

パワフル・おやすみ、それぞれON/OFFを個別に単独操作で往復させ、他バイトが一切変化しないことを
確認した。**byte[13]は複数の機能を独立ビットで持つフラグバイト**とみられる。

| 機能 | bit | 値 |
|---|---|---|
| パワフル | `0x01` | OFF=`0x00` / ON=`0x01` |
| おやすみ | `0x04` | OFF=`0x00` / ON=`0x04` |

(パワフルON時に風量/温度の液晶表示が消える、おやすみON時は「おやすみ〇〇」表示になる、という
UI変化はあるが、送信データ上はbitが立つだけで風量バイト等は元の設定値のまま送られ続ける)

→ 新しく以下をcomponent側に実装できる(まだコード未反映):
```cpp
constexpr int DAIKIN_POWERFUL_BYTE_INDEX = 13;
constexpr uint8_t DAIKIN_POWERFUL_BIT_MASK = 0x01;
constexpr int DAIKIN_SLEEP_BYTE_INDEX = 13;
constexpr uint8_t DAIKIN_SLEEP_BIT_MASK = 0x04;
```
他のbit(0x02, 0x08, 0x10, 0x20, 0x40, 0x80)は今のところ全て0のまま観測されている。
エコノ的なボタンはこのリモコンに存在しないとのことなので、このバイトについては
これで実用上十分と考えられる。

### 2-5. 風向・スイング — **未解明(重要)**

風量とは違い、風向上下・風向左右・センサー風向ボタンを押しても **state frame の byte[8]下位nibble /
byte[9] は実測範囲では一貫して `0x00` のまま変化しなかった**(byte[8]は風量の値がそのまま、
下位nibbleは常に0)。既存コンポーネントの`fan_speed_()`は
`swing_mode`に応じて`byte[8]|=0x0F`(垂直)/`byte[9]|=0x0F`(水平)するロジックになっているが、
**ARC472A66の実機ではこの仕様が効いていない可能性が高い**。

代わりに、風向ボタンを押すと **header frame (20バイト)側の byte[9]・byte[13] が変化する**
ことを確認した:

| タイミング | header byte[9] | header byte[13] | 備考 |
|---|---|---|---|
| 冷房ON直後(風向未調整) | `0x0e` | `0x28` | |
| 風向ボタンを何度か押した後 | `0x16` | `0x3f`→`0x3e` | 「センサー自動」に調整した後 |
| 停止時 | `0x02` | (未記録) | 冷房→停止・ドライ→停止どちらも`0x02` |
| パワフルON(風向表示が上下自動/左右センサー自動になった) | `0x19` | `0x3e` | 風向ボタンは押していない。パワフル側の副作用で表示が変わった可能性 |

→ **仮説**: このリモコン(ARC472A66)は風向コマンドを19バイトのstate frameではなく、
20バイトのheader frame側に載せている可能性が高い。ただし header byte[5] は同じ論理状態でも
セッションをまたぐと値が変わる(`0x6e`→`0x74`→`0x76`→`0x88`→`0x8b`→`0x8c`→`0x8e`のように、
同一状態内でもパワフルON/OFFの前後だけで微妙に増減している)ため、**部屋の温度センサー値や
何らかのカウンタが混ざっている**とみられ、素直な固定ビットではなさそう。

風向ボタンは3種類あるとの申告:
- センサー風向ボタン
- 風向上下ボタン
- 風向左右ボタン

**【解決・2026-07-19訂正】風向上下・風向左右とも完全に解読できた**(header frameの別バイトで、
単独ボタンの1周テストにより確定):

**【重要・訂正】風向上下は byte[11] ではなく byte[12] だった**。当初「風向上下 = header
byte[11]」と記録していたが、これはバイトの数え間違い(オフバイワン)だったことが2026-07-19の
自動復帰設定の実機検証で判明した。byte[11]は本節の値ではなく、実際には**自動復帰設定**の
バイトだった(8章参照)。以下の表の値自体(0x16など)は正しいが、位置がbyte[12]である点に
注意。既存の`daikin_arc472a66.cpp`が`remote_header[11]`に風向上下の値を書き込んでいたのは
このオフバイワンが原因で、結果的に自動復帰設定のバイトを意図せず上書きし続けていたことが
「常に自動復帰が30分になる」不具合の正体だった。5章のアーカイブデータも、実は全て
byte[11]=`0x30`(自動復帰オフ)で一貫しており、風向上下の値は一貫してbyte[12]側に出ている
(例: 5-12節「元の固定角度(上=16, 左右=28)に復帰」の生データを見るとbyte[12]=`16`,
byte[13]=`28`になっている)。

**センサー風向ボタン = state frame byte[16] の3値サイクル + header両軸を自動化**

センサー風向ボタンを押すと、header側の上下(byte12)・左右(byte13)が**同時に自動値
(`0xE6`/`0x3E`)にセットされ**、さらに**state frame側のbyte[16]が`0x80(OFF)→0x83(エリア)
→0x84(スポット)→0x80(OFF、元の固定角度に戻る)`と3状態でサイクルする**ことを確認した
(これまでbyte[16]は「常に0x80固定」と誤認していたが、センサー風向機能だけの例外)。

| 状態 | byte[16] |
|---|---|
| OFF(センサー風向オフ、通常の固定/自動角度に戻る) | `0x80` |
| エリア(広めの範囲を人感センサーで検知) | `0x83` |
| スポット(ピンポイントで人感センサーが検知) | `0x84` |

OFFに戻ると、header側のbyte12/13は「センサー風向を押す直前に設定していた固定角度」に
戻る(自動には戻らない)。連番ではない値(`0x83`/`0x84`)なので単純ビットマスクではなく、
3値の列挙型として扱うのが良さそう。

補足: header byte[9]は「風向上下ボタンを一度でも押すと`0x0e→0x06`に変わってそのまま」という
挙動が見られた一方、風向左右だけを操作したテストでは終始`0x16`で不変だった。byte[9]は
おそらく上下方向の別の状態フラグ(あるいは単に電源ON直後だけ`0x0e`になる一時的な値)と
考えられ、実用上はbyte[11]/byte[13]の2つを追えば十分。

→ **結論**: 風向上下・風向左右・センサー風向・サーキュレーションの4ボタンは、いずれも
19バイトのstate frameには一切影響せず、20バイトのheader frame側(byte[9]/[12]/[13]あたり)
だけで表現されている。今のコンポーネントはheader frameを固定値でしか送っていないので、
**これらの機能をESPHomeのswitch/select等で公開したい場合は、header frameを動的に
組み立てるロジックを新規に書く必要がある**(現状のclimateの`swing_mode`3値を
state frameのbit操作にマッピングする今の実装は、ARC472A66では機能しない)。

### 2-6. 内部クリーン・フィルター掃除 — **【重要】19バイトstate frameとは別の8バイト特殊フレーム**

README冒頭からの最大の目的だった「内部クリーンに入らない問題」について、**前提が間違っていた
ことが判明した**。内部クリーンは停止フレームに隠れたビットではなく、**内部クリーン専用ボタン
(2秒長押し)が送る、19バイトとは全くの別物の8バイト短縮フレーム**だった:

```
[内部クリーン(2秒長押し)]
11 da 27 00 84 0c 00 a2

[フィルター掃除(1回押し)]
11 da 27 00 84 14 00 aa

[お知らせ(1回押し)]
11 da 27 00 84 87 20 3d
```

構造: 固定ヘッダ`11 DA 27 00` + `0x84`(特殊コマンド種別) + サブ機能コード(内部クリーン`0x0c`/
フィルター掃除`0x14`/お知らせ`0x87`) + 1バイト(内部クリーン・フィルター掃除は`0x00`固定、
お知らせだけ`0x20`) + チェックサム(単純8bit加算、他フレームと同じ方式)。
内部クリーン・フィルター掃除は押した瞬間だけ表示が変わり、その後は通常の停止画面に戻る
(=単発トリガー的な「開始コマンド」であって、状態を保持するビットではない)。

**お知らせボタンを押すと「受信失敗、もう一度」と表示された**。これは実機エアコン(または
このIRスニファ)からの応答が返ってこなかったためと考えられる。おそらく「お知らせ」は
エアコン側にフィルター状況やエラー情報などを問い合わせ、双方向のやりとりで応答を待つ
仕様になっており、この検証環境ではエアコンからの返信をシミュレートしていないため
失敗表示になったと推測される(実機のエアコンに向けて押せば正常に応答が返る可能性が高い)。
コマンド自体は正しく送信できているので、リモコンコードとしては解析完了とみなせる。

**分かったこと・意味合い**:
- はおりんさんの申告どおり「ある程度運転した後に停止すると自動で内部クリーンに入ることがある」
  というのは、**リモコンが何か特別なコマンドを送っているのではなく、エアコン本体が
  コンプレッサー稼働時間等から自律的に判断している**とみられる。つまりREADMEに書いてあった
  「停止フレームのbyte[5]/byte[6]を直前モードのまま残す」修正(4章、実装済み)だけで、
  本来この自律判断のロジックには正しく載っているはずで、**追加のビット探しは不要だった**
  可能性が高い。
- 「確実に内部クリーンを開始させたい」場合は、`power_off_internal_clean`オプションで
  停止フレームにビットを足す方式ではなく、**この8バイトの内部クリーン専用フレームを
  停止フレームの直後に追加送信する**方式に作り直すのが、実機の動作に一番忠実。
  (フィルター掃除も同じ方式でESPHome側にボタン/スイッチとして追加できる)
- `daikin_arc472a66.h`の`INTERNAL_CLEAN_BYTE_INDEX`/`INTERNAL_CLEAN_BIT_MASK`という仕組み自体
  (state frameにビットを足す方式)は、この実測結果からは**根拠がなくなった**。実装し直すときは
  この8バイトフレーム送信に置き換えるべき。

**TODO(未実施)**: フィルター掃除・内部クリーン以外にも同系統の短縮フレームを送るボタンが
ないか(お知らせボタンなど)。優先度は低い。実装(コード反映)は解析が一通り終わってから
まとめて行う方針(はおりんさんの指示)。

## 3. header frame (20バイト) のレイアウト

| idx | 内容 |
|---|---|
| 0-3 | 固定ヘッダ `11 DA 27 00` |
| 4 | 常に `0x02` |
| 5 | **ローリングカウンタ(意味なし)**。同一状態でも送信ごとに+1前後で単調増加していく。ボタン識別には無関係、無視してよい |
| 6 | 電源ON中は`0x44`、停止中は`0xC4`(bit0x80=停止フラグとみられる) |
| 7 | 常に `0x02` |
| 8 | 常に `0x20` |
| 9 | **風向系ボタン(風向上下/左右/センサー)で変化**。冷房ON直後`0x0e`、調整後`0x16`、停止時`0x02`、パワフルON時`0x19`。個別の風向ボタンの切り分けは未実施(TODO) |
| 10 | 常に `0x83` |
| 11 | **自動復帰設定・確定(2026-07-19実機検証)**: オフ=`0x30` / 30分=`0x20` / 1時間=`0x10`。モードには依存しない(8章参照)。旧版ではここに「モードによって0x30⇔0x36」と誤記していたが、それは実際にはbyte[12](風向上下)の値を見誤っていたもの |
| 12 | **風向上下 = 2-5節参照(旧版ではbyte11と誤記)**。サーキュレーションONだとbit0x10が落ちる(AUTO: `0xE6`→`0xD6`) |
| 13 | **風向系ボタンで変化**。`0x28`(初期)→`0x3f`→`0x3e`。9番と同様、個別切り分けは未実施 |
| 14 | **送風(空清)ボタン=Streamer ON/OFFで変化・ほぼ確定**: bit`0x10`が Streamer ON=`0xd2` / OFF=`0xc2` |
| 15 | 常に `0xf0` |
| 16 | 常に `0x00` |
| 17 | 常に `0x36` |
| 18 | 常に `0x00` |
| 19 | チェックサム |

現行 `daikin_arc472a66.cpp` の `transmit_state()` は、この header frame を
**送信時は常に固定値**で送っている(`remote_header[]`の初期値のまま、湿度変更時に`header[9] |= 0x10`
するだけ)。実測データはあくまで**純正リモコンからの受信ログ**なので、送信側がこの固定値のままで
AC側が正しく動くかどうかは別問題(README Step4の「実機で確認」がまだ)。
風向・サーキュレーション・Streamerを本格的にESPHome側から制御したい場合は、ここをちゃんと
動的に組み立てる必要がある。

### 3-1. 注意: 別の「本家daikin_arc」デバイスからの混信

解析中に一度、**このプロジェクトのコンポーネントではない別のESPHome/HA機器**(はおりんさん曰く
「スマートリモコン」)がこの実機エアコンに命令を送り、たまたまキャプチャに混ざったことがある。
見分け方: state frameのbyte[17]が本来`0x2a`(ARC472A66実測値)のはずが`0x24`(本家daikin_arc
未修正版のデフォルト)になっている、header frameが完全固定の
`11 da 27 00 02 d0 02 03 80 (03|13) 82 30 41 1f 82 f4 00 24 00 xx`になっている、という
2点で判別できる。**この形が出てきたら自分の実機リモコン操作ではないので無視してよい。**
(家に別のダイキンエアコン制御用スマートリモコンが存在する可能性があるとのこと。動作upに支障は
なさそうだが、混信で解析データが汚れる可能性があるので今後も注意する)

## 4. 停止(電源OFF)時の挙動 — 確定・実装済み

README/コード内コメントの通り: 停止ボタンを押しても `byte[5]`(モード)・`byte[6]`(温度)は
OFF専用の固定値に潰されず、**直前のモードのビットを残したまま電源ビットだけ落とす**。
`daikin_arc472a66.cpp`の`last_active_mode_`メンバで実装済み。**内部クリーンに入るかどうかの
実機確認(README Step4)はまだ行っていない。**

### 実装タスク一覧 (優先度順)

1. **`daikin_arc472a66.h` のコメント更新**
   - `DAIKIN_MODE_*` / `DAIKIN_FAN_*` の「TODO: 実測して確認」を削除し、2-1・2-3節の実測結果で
     確定済みである旨に書き換える(値そのものは変更不要、既存定数のままでOK)。

2. **パワフル・おやすみをコンポーネントに追加** (2-4節)
   - `remote_state[13]` に `DAIKIN_POWERFUL_BIT (0x01)` / `DAIKIN_SLEEP_BIT (0x04)` を定義。
   - climate_ir の標準traitに専用スイッチはないので、`switch`コンポーネントとして
     `set_powerful()`/`set_sleep()`のようなsetterをclimateクラスに生やし、`transmit_state()`
     内で`remote_state[13] |= ...`する形が素直(本家daikin_arcの`powerful_`実装が参考になる)。

3. **内部クリーン/フィルター掃除の実装方式を刷新** (2-6節)
   - 今の`INTERNAL_CLEAN_BYTE_INDEX`/`INTERNAL_CLEAN_BIT_MASK`(state frameにビットを足す方式)
     は根拠がなくなったので**削除**する。
   - 代わりに、8バイト特殊フレーム `11 DA 27 00 84 <sub> 00 <checksum>` を送る関数
     (例: `send_special_command_(uint8_t sub_code)`)を新設し、
     `set_power_off_internal_clean(true)`のときは**停止フレーム送信直後にsub_code=0x0cを追加送信**
     する形に作り直す。フィルター掃除(`0x14`)も同じ関数で`button`コンポーネントとして追加できる。
   - README Step4(実機での内部クリーン動作確認)は、まずは「何もしなくても直前モード保持だけで
     自動的に内部クリーンに入るか」を確認し、ダメなら8バイトフレーム追加送信を試す、の順で。

4. **温度式・湿度式をclimateロジックに反映**
   - 冷房/暖房: 既存の式のままでOK(2-2節で確認済み)。
   - 快適エコ自動(`CLIMATE_MODE_HEAT_COOL`)の温度: 今は`0xC0`固定を返しているが、
     2-2節の式(`n>=0: 0xC0+n*2`, `n<0: 0xE0+n*2`)を使えば`target_temperature`を
     オフセットとして解釈し温度調整に対応できる。ただし対応範囲(±何度まで)が未確認なので、
     実装するなら安全マージンを取るかクランプすること。
   - 除湿の湿度(2-6節手前、byte[7]確定分): 生%値をそのまま送るだけなので
     既存の`humidity_()`をシンプルな`return (uint8_t) this->target_humidity;`的な形に
     変更できる可能性がある(現状の40/45/50固定値ロジックより実態に近い)。

5. **風向・サーキュレーション・Streamer(header frame制御)は対応するかどうかを検討**
   - 3章・2-5節の通り、これらはすべて20バイトheader frame側の機能で、今の実装はheaderを
     固定値でしか送っていない。対応するなら`transmit_state()`でheaderを動的に組み立てる
     大きめの変更が必要(byte9/12/13が絡む。byte11は自動復帰設定なので触らない)。
   - 費用対効果が低ければ「非対応」と明記して見送るのも妥当な判断(既存の`swing_mode`
     3値ベースの実装は今のARC472A66では効かないので、いずれにせよ現状のロジックのままでは
     動かない=どのみち触る必要はある)。

## 8. 自動復帰設定 (header byte[11]) — 確定・実装済み (2026-07-19)

リモコンのメニュー内から設定する「自動復帰設定」(電源ON操作から一定時間で自動的に運転を
止める機能。オフ/30分/1時間の3段階)が、ESPHomeコンポーネント経由で操作すると常に「30分」
相当の挙動になってしまう、という報告を受けて実機検証した。

**検証方法**: 冷房・28℃・風量を完全に固定した状態で、自動復帰設定だけを
オフ→30分→1時間と変えながら、都度「停止→電源ON」を1回ずつ送信・キャプチャして比較した。

**実測結果 (冷房28℃・風量しずか・電源ON時のheader frame)**:

```
[自動復帰オフ]
header: 11 da 27 00 02 a8 42 02 20 0e 83 30 16 28 c2 f0 00 36 00 07

[自動復帰30分]
header: 11 da 27 00 02 a9 42 02 20 0e 83 20 16 28 c2 f0 00 36 00 f8

[自動復帰1時間]
header: 11 da 27 00 02 ab 42 02 20 0e 83 10 16 28 c2 f0 00 36 00 ea
```

byte[5](ローリングカウンタ、無視してよい)以外で変化したのは **byte[11]のみ**:
`0x30`(オフ) → `0x20`(30分) → `0x10`(1時間)。state frame側は3パターンとも完全に無変化。

暖房モードでの古いアーカイブデータ(5-5節、自動復帰は当時オフだったはず)でも
byte[11]は`0x30`のままだったため、**モードには依存しない**と判断した。

**結論**:
```cpp
enum class AutoReturn : uint8_t {
  OFF = 0x30,
  THIRTY_MINUTES = 0x20,
  ONE_HOUR = 0x10,
};
```
`remote_header[11] = static_cast<uint8_t>(auto_return_);` を`transmit_state()`に実装済み。
YAML側は `auto_return: OFF / THIRTY_MINUTES / ONE_HOUR` で選択可能(デフォルト`OFF`)。

**副産物としての訂正**: この検証中に、2-5節で「風向上下 = byte[11]」としていた記録が
実際には**byte[12]の誤り**(オフバイワン)だったことが判明した。既存コードが
`remote_header[11] = vertical_vane_byte_();` としていたため、風向上下の値(例:
`0x16`=位置1固定)で自動復帰設定のバイトを意図せず上書きしてしまい、それが今回の不具合の
直接原因だった。あわせて修正済み(`remote_header[12]`に変更、サーキュレーションのビット
合成もそちらに統合)。詳細は2-5節・3章の訂正箇所を参照。
